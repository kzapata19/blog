A few weeks ago I paired up with Jeremy to help jump start my Ruby Tic-Tac-Toe using TDD. About two weeks prior I had never programmed in Ruby and had only a vague idea about what this TDD buzzword meant. I did however have plenty of experience playing Tic-Tac-Toe :)

Personally, I tend to feel more comfortable working in very small groups than larger ones so I naturally welcomed the idea to pair program.

### The Setup

One computer, one monitor, one keyboard. One person drives (types the code) and the other navigates (reviews the code and provides guidance). Periodically both people switch roles. In addition to this setup, the game had to be designed using TDD so we switched roles every time we created a new unit test.

After creating and passing our first unit test, I realized that combining pair programming and TDD ensured that the tests were testing the right thing.

The value of having someone else working directly with you on your code cannot be overstated. Simply voicing my ideas out loud helped streamline my jumbled up thoughts and immediately expose any knowledge gaps and assumptions. Also, I've read that the human brain can only retain about six to eight ideas at a time. So with two people that doubles the ideas we can keep track of at once. An added benefit I noticed was that we also took turns keeping track of the bigger picture and the specific issue at hand.

Since that pair programming experience, I've taken over the Tic-Tac-Toe game solo. These past two weeks I've noticed that I've found it harder to debug my code. I definitely suspect that not having an opportunity to discuss my ideas out loud plays a role but I also think that my approach to debugging solo takes a different form. I noticed that debugging solo entails getting lost in the details and implementing brute force attempts. I need to find a way to simulate the pair programming experience in a way that helps me gain a more high level perspective.

I've heard talking to a rubber ducky helps.
